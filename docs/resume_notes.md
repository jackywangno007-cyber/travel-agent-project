# 简历与面试说明

## 一版适合春招简历的项目描述

### 版本 1：适合写进项目经历

智能旅行助手 Travel Agent Project  
独立完成一个基于 Vue3 + FastAPI + LLM 的旅行规划系统，围绕真实出行场景实现了景点、天气、酒店、餐饮、交通、地图、预算、导出和单日重规划等能力。后端采用多 Agent + 编排层结构，接入高德 POI、天气服务和大模型接口，并通过 schema 校验、规则后处理与 fallback 机制提升结果稳定性；前端实现了地图联动、预算拆分、行程编辑、图片/PDF 导出等交互能力，形成了从需求输入到可执行旅行计划输出的完整闭环。

### 版本 2：适合压缩成两三行

基于 Vue3、TypeScript、FastAPI 与大模型接口开发智能旅行助手，接入高德 POI、真实天气与地图能力，支持多日行程生成、预算拆分、地图联动、导出和单日重规划。通过多 Agent 编排、规则后处理和 fallback 机制提升规划结果稳定性与可执行性。

- 前端支持编辑、地图联动、预算重算、单日重规划，交互完整度高
- 项目架构从直接 service 调用逐步抽象到共享 tool provider，具备继续演进为 MCP 风格的基础
- 项目关注“工程完整性”，不仅有生成，还覆盖了导出、展示、异常处理和局部更新

## 面试时推荐的讲述结构

### 1. 先讲业务目标

我做的是一个智能旅行助手，不只是给一段推荐文本，而是希望输出一份真正可执行的旅行计划。

### 2. 再讲核心技术路线

后端用 FastAPI 做接口编排，把景点、天气、酒店、餐饮这些能力拆成独立 agent；planner 再结合大模型和规则后处理生成 daily plan。前端用 Vue3 展示地图、预算和每日行程，并支持编辑和单日重规划。

这个项目让我把“外部工具接入 + 大模型规划 + 前端交互 + 工程化落地”完整串了一遍，也为后续继续扩展历史记录、路线规划和 MCP 风格工具调用打下了基础。
# 春招简历项目描述

## 1. 一版适合直接放简历的项目描述

**智能旅行助手 Travel Agent（个人项目）**  
基于 Vue3 + TypeScript + FastAPI + 多 Agent 架构实现智能旅行规划系统，接入高德 POI、真实天气和大模型能力，支持根据用户输入自动生成包含景点、酒店、餐饮、交通、地图和预算的多天旅行计划；进一步实现了结果页地图联动、预算拆分、景点编辑、导出 PDF/图片以及单日重规划能力，形成较完整的交互式 Agent 产品闭环。

---

## 2. 简历亮点拆分写法

- 设计并实现前后端分离的智能旅行规划系统，支持多天行程自动生成与结果页交互编辑。
- 接入高德 POI、真实天气、大模型规划能力，打通景点、餐饮、酒店、交通、预算等多维数据链路。
- 通过 `planner_agent + 规则后处理` 提升 LLM 输出稳定性，解决天数不一致、景点重复、预算异常等问题。
- 实现地图打点、预算拆分、PDF/图片导出、单日重规划等功能，增强产品可用性与展示完整度。
- 抽象共享 Tool Provider，为后续 MCP 风格工具接入预留架构演进空间。

---

## 3. 面试可重点讲的 5 个亮点

### 亮点 1：不是纯调用 LLM，而是“LLM + 工具 + 后处理”

可重点讲：

- 大模型负责生成行程框架
- 真实景点、天气、酒店、餐饮来自工具层
- 结果返回前再做规则校验和修正

### 亮点 2：做了真实产品交互，而不只是接口

可重点讲：

- 地图联动
- 景点删除与排序
- 单日重规划
- 导出图片 / PDF

### 亮点 3：有稳定性兜底

可重点讲：

- 高德失败会 fallback
- 天气失败会 fallback
- 餐饮失败也不会空
- LLM 输出不合法会走规则回退

### 亮点 4：有架构意识

可重点讲：

- API / orchestrator / agents / services / tools 分层
- 共享 Tool Provider 的设计
- 渐进式向 MCP 风格演进，而不是一次性重构

### 亮点 5：有前后端联动的数据闭环

可重点讲：

- DayPlan 是统一核心数据结构
- 删除景点、排序、重规划后，地图和预算都能联动更新

---

## 4. 面试时推荐讲述顺序

### 第一步：先讲产品目标

“我想做的不是一个只返回文字的 Demo，而是一个更接近真实用户可操作的旅行助手。”

### 第二步：讲整体架构

“前端负责输入、展示、编辑和导出；后端负责编排景点、天气、酒店、餐饮等多个 agent，再由 planner 统一生成行程。”

### 第三步：讲最有代表性的技术难点

- 大模型输出不稳定，如何做后处理
- 日期和 daily_plan 如何严格对齐
- 如何让地图、预算、编辑围绕同一份数据联动
- 如何做单日重规划，而不是重跑整份行程

### 第四步：讲工程取舍

“我没有一开始就全量上复杂路线 API 或完整 MCP，而是先把系统跑通，再逐步抽象工具层，控制复杂度和稳定性风险。”

### 第五步：讲可继续扩展的方向

- 历史记录
- 更细的路线规划
- 更多用户偏好控制
- MCP 化工具接入
